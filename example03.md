## 实现思路拆解

**问题 1：避免组件臃肿**

→ 可以采用逻辑与视图分离的方式：

渲染逻辑抽离到独立的Logic.ts文件中，视图组件仅专注于渲染，实现 "单一职责"。

example03_2中的ShopViewLogic.ts和ShopView.vue就是采用这种方案的。

好处是：当需求变更时（如需修改逻辑），只需修改Logic.ts而不触碰视图代码；同时视图调整也不会影响业务逻辑，一定程度上有利于维护。

**问题 2：状态管理**

→ 轻量级状态管理方案：

example03 没有采用 Vuex/Pinia 等全局状态库，而是通过@vueuse/core的createGlobalState创建全局状态，配合ref和shallowRef实现响应式。

这种方案的优势在于：既满足了跨组件状态共享（如订单数据在ShopView和OrderCost间共享），又避免了全局状态库带来的概念和配置负担，对于中小型项目更为轻量高效。

example03_2中采用的是自定义store，是一种灵活的状态管理方案，相比使用全局状态库，它的好处是轻量灵活，比较适合小型项目，较大的项目中全局状态的优势更能体现。

**问题 3：shallowRef与ref**

→ 在响应式状态选择上，shallowRef 和 ref 各有适用场景，不应盲目全用 ref：：

ref 适合深层响应场景：当数据是复杂对象（如包含嵌套属性的 Shop、Order），且需要监听内部属性变化时（如修改商品价格、订单数量），用 ref 会递归处理所有属性，确保深层变化能触发视图更新。

shallowRef 适合浅层次响应：当数据是简单值（如字符串、数字）或 “只读 / 极少修改内部属性” 的复杂对象（如店铺列表 Shops[]，仅整体替换而不频繁修改单个属性），用 shallowRef 仅监听顶层引用变化，避免递归响应带来的性能损耗，更高效。

简言之：需要深层响应 → 用 ref；仅需顶层响应或数据结构稳定 → 用 shallowRef，按需选择才能兼顾响应性和性能。

**问题 4：**

→ onMounted 和 .then 是不同场景下处理异步逻辑的方式：

onMounted：Vue 生命周期钩子

作用：在组件挂载完成后（DOM 已渲染）执行回调函数，用于初始化操作（如获取数据、绑定事件等）。

适用场景：与组件生命周期强相关的逻辑，确保操作在 DOM 就绪后执行。

.then：Promise 链式调用

作用：用于处理 Promise 对象的异步结果，在 Promise 状态变为 resolved 时执行回调。

适用场景：任何返回 Promise 的异步操作（如接口请求、定时器等），与组件生命周期无关。
